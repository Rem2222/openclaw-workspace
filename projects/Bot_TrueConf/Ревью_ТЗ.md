## 3. Проверка рисков и ограничений

Проверено против `План_разработки.md` → секция "Риски".

| Риск из Плана | Отражён в ТЗ | Статус |
|---------------|--------------|--------|
| python-trueconf-bot API нестабилен (v1.2.x) | Да, в "Ограничения" этапа 7 и приложение Б | ✅ |
| Gateway polling vs WebSocket | Частично — ТЗ выбрал sync HTTP, но нет плана отказа если streaming не заработает | 🟡 |
| Файловые лимиты TrueConf | Да, в "Ограничения" этапа 8 (10 МБ) | ✅ |

### Дополнительные риски, НЕ учтённые в ТЗ 🔴

1. **Rate limit auth Gateway** — `gateway.auth.rateLimit` → 429 с `Retry-After`. Бот должен уметь обрабатывать 429 при bursts. **Не описано в ТЗ.**
2. **Operator-level security boundary** — документация прямо пишет:  
   > Treat this endpoint as a **full operator-access** surface.  
   > Keep this endpoint on loopback/tailnet/private ingress only.  
   
   В ТЗ не указано, что Gateway **нельзя** выставлять наружу и нужно bind на loopback/tailnet. Это критично для прод-деплоя (этап 13).
3. **Timeout согласованность** — в SPEC п.5.1 указан gateway timeout **60 сек**, а в ТЗ этап 6A — **120 сек**. Противоречие.
4. **Лимит `maxUrlParts: 8`** для `/v1/responses` (общий лимит URL-based input_file + input_image на запрос) — не упомянут в ТЗ.
5. **PDF behavior** — в `/v1/responses` PDFs могут рендериться в изображения при малом количестве текста. Это может изменить поведение и размер запроса. Не описано.
6. **`input_file` контент попадает в system prompt как untrusted external content** — важное архитектурное отличие от multimodal message content. Бот должен понимать, что файл **не становится частью user-сообщения**, и это влияет на UX (агент "видит" файл иначе).

## 5. Замечания

### Критические 🔴

1. **FR-15 (выгрузка файлов агентом пользователю) не имеет конкретной реализации.**  
   **Что делать:** Определить перед этапом 8 один из путей:  
   (a) `/v1/responses` streaming + агент возвращает base64 в тексте ответа + бот парсит → отправляет файл пользователю;  
   (b) Агент сохраняет файл в известный каталог, бот делает `fs_read` через `/tools/invoke` (учесть, что `fs_write` в deny-list, но `fs_read` — нет);  
   (c) Отдельная out-of-band шина (например, через общий tmpfs).  
   Без этого этап 8 заблокирован.

2. **Timeout противоречит между SPEC (60 сек) и ТЗ (120 сек).**  
   Выбрать одно значение и синхронизировать. Рекомендация: 120 сек (совпадает с `files.timeoutMs` / reasonable для agent turns).

3. **Security boundary не отражена в ТЗ.**  
   В этапе 13 (Deploy) необходимо явно прописать:  
   - `gateway.auth.mode = "token"` + надёжный `OPENCLAW_GATEWAY_TOKEN`  
   - Gateway bind only on loopback (`127.0.0.1`) или tailnet  
   - Бот и Gateway **на одном хосте** (идеально) или через защищённую сеть  
   - Никогда не выставлять `:18789` на публичный IP

4. **SPEC.md содержит несуществующие endpoints (`/sessions.send`, `/sessions.create` и т.п.).**  
   Это не блокирует ТЗ (v1.1 правильно обошёл), но **SPEC должен быть исправлен** для будущих читателей, либо в ТЗ явно зафиксировано "SPEC п. 6.2 устарел, актуальная архитектура — см. ТЗ Приложение А".

### Важные 🟡

5. **FR-10 (tool indicators) — не определён контракт.**  
   `/v1/chat/completions` OpenAI-streaming не содержит custom tool events из OpenClaw. `/v1/responses` содержит `response.output_item.added` — но неизвестно, будут ли там tool calls агента (в документации они описаны только для **client-side function tools**, не для внутренних). Предлагаю перед этапом 10 провести spike-тест (30 мин): сделать реальный запрос с streaming, посмотреть события. Без spike — этап 10 рискует уйти в fallback.

6. **FR-18 обработка отвала сессии** — не выделено в отдельной секции ТЗ. Рекомендация: добавить в этап 6A задачу "обработка 401/404/500 от Gateway + user-friendly error message".

7. **Rate limit на auth Gateway (429 + Retry-After)** — добавить в этап 9 как отдельный пункт rate-limit handling.

8. **Противоречие streaming vs sync** — этап 6A пишет "синхронный (не streaming) — ждём полный ответ", а этап 10 полагается на streaming. Нужно: этап 6A должен поддерживать **оба режима** (stream=False для простого текста, stream=True для tool indicators).

9. **`/v1/chat/completions` multimodal для TrueConf files (этап 8)** — нет достоверных данных, что OpenClaw поддерживает OpenAI `image_url` в `/v1/chat/completions`. В документации этот путь описан **только для `/v1/responses`**. Рекомендация: **убрать "Вариант A"** из этапа 8, оставить **`/v1/responses`** как основной путь для файлов.

### Рекомендации 🟢

10. **Выбор session routing** — зафиксировать явно: `x-openclaw-session-key` (не OpenAI `user`). Это упрощает отладку.

11. **SSL verify** — в SPEC 5.2 написано "dev — False, prod — True". Добавить в этап 13 чек-лист: убедиться что `ssl_verify = true` перед деплоем.

12. **Логирование** — в этап 5 добавить формат логов (JSON? plain?), уровни (INFO для incoming, WARN для 429, ERROR для gateway failures).

13. **Versioning** — добавить в этапе 12 (Docs) упоминание версии API Gateway, под которую писали бот (чтобы при обновлении OpenClaw знать, что проверять).

14. **Health check** — не описан в ТЗ endpoint `/health` бота (для systemd watchdog / мониторинга). Рекомендация: добавить в этап 13.

15. **Тесты для FR-15 (файл от агента)** — пока FR-15 не реализован, тесты для него из этапа 11 тоже зависают.
