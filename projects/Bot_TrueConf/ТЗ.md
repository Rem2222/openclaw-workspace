## Общая информация

Документ описывает техническое задание для каждого этапа разработки бота. Этапы соответствуют плану из `План_разработки.md`.

### Важное замечание по API

**`sessions_send` заблокирован в HTTP `/tools/invoke`** (default deny list). Бот должен использовать **OpenAI-совместимый `/v1/chat/completions`** endpoint Gateway для отправки сообщений агенту. Это потребует включения `gateway.http.endpoints.chatCompletions.enabled = true` в конфигурации Gateway.

*Источник: `/usr/lib/node_modules/openclaw/docs/gateway/tools-invoke-http-api.md` — раздел "Policy + routing behavior", deny list включает `sessions_send`.*

*Источник: `/usr/lib/node_modules/openclaw/docs/gateway/openai-http-api.md` — `/v1/chat/completions` как основной HTTP-интерфейс для чата с агентом.*

## Этап 6A: Gateway-клиент (HTTP + sessions)

### Цель
Реализовать HTTP-клиент для взаимодействия с OpenClaw Gateway.

### Задачи
1. Реализовать `app/gateway/client.py`:
   - `GatewayClient` с `httpx.AsyncClient`
   - Авторизация: `Authorization: Bearer <token>`
   - Базовый URL: `http://<host>:<port>`
2. Реализовать `app/gateway/sessions.py`:
   - `send_message(session_key, text) → str` — отправка через `/v1/chat/completions`
   - `create_session(user_id) → session_key` — генерация ключа `trueconf_{user_id}`
   - `reset_session(session_key)` — сброс сессии (если доступен через API)
3. Реализовать `app/session_store.py`:
   - Хранение `user_id → session_key` в памяти (dict)
   - TTL-таймер: 3 часа idle → сброс
   - Метод `get_or_create(user_id) → session_key`

### API Gateway — подтверждённые endpoints

| Endpoint | Метод | Назначение | Источник |
|----------|-------|-----------|----------|
| `/v1/chat/completions` | POST | Отправка сообщения агенту, получение ответа | `openai-http-api.md` |
| `/v1/models` | GET | Список доступных моделей/агентов | `openai-http-api.md` |
| `/tools/invoke` | POST | Вызов tools (ограничен deny list) | `tools-invoke-http-api.md` |

**Важно:** `/v1/chat/completions` должен быть включён в конфигурации Gateway:
```json5
{ gateway: { http: { endpoints: { chatCompletions: { enabled: true } } } } }
```

**Отправка сообщения:**
```python
# POST /v1/chat/completions
headers = {
    "Authorization": f"Bearer {token}",
    "x-openclaw-session-key": session_key,  # trueconf_{user_id}
    "Content-Type": "application/json"
}
body = {
    "model": "openclaw",  # default agent
    "messages": [{"role": "user", "content": text}],
    "stream": false
}
```

**Получение ответа:** синхронный (не streaming) — ждём полный ответ.

### Критерии приёмки
- `GatewayClient` инициализируется с host/port/token
- `send_message()` отправляет POST на `/v1/chat/completions` с правильными заголовками
- `SessionStore` хранит сессии с TTL 3 часа
- Unit-тесты: mock httpx, проверить формирование запроса

### Ожидаемые файлы
- `app/gateway/client.py`
- `app/gateway/sessions.py`
- `app/session_store.py`
- `tests/test_gateway.py`

### Технологии
- `httpx` — async HTTP клиент
- `unittest` / `pytest`

### Ограничения
- Не использовать `/tools/invoke` для `sessions_send` (заблокирован)
- Timeout на запрос: 120 сек
- Ответ агента может быть длинным — обрабатывать chunked responses

## Этап 7: Обработчики — текст, команды, reply

### Цель
Реализовать обработчики входящих сообщений от TrueConf.

### Задачи
1. `app/bot.py` — инициализация `Bot`, `Dispatcher`, `Router`:
   ```python
   bot = Bot.from_credentials(username, password, server)
   dp = Dispatcher()
   router = Router()
   dp.include_router(router)
   ```
2. `app/handlers/text.py`:
   - `@router.message(F.text)` — ловит все текстовые сообщения
   - Проверка allowlist (`allowed_users.py`)
   - Отправка "Барыга ищет товар... 🕵️" через `msg.answer()`
   - Вызов `gateway.sessions.send_message(session_key, text)`
   - Конвертация ответа Markdown → HTML
   - Редактирование placeholder: `message.edit_text(answer, parse_mode=ParseMode.HTML)`
3. `app/handlers/commands.py`:
   - `/start`, `/help`, `/status` — отправляются агенту как обычный текст
   - `/reset` — вызов `session_store.reset()` + ответ "🔄 Сессия сброшена"
4. `app/handlers/replies.py`:
   - `@router.message(F.reply_to_message)`
   - Извлечение `msg.reply_to_message.text`
   - Добавление контекста: "Пользователь отвечает на: {original_text}\n\n{new_text}"
5. `app/allowed_users.py`:
   - Загрузка списка из `config.toml` → `[access].allowed_users`
   - Проверка: `is_allowed(username: str) → bool`

### Критерии приёмки
- Текстовое сообщение пересылается агенту и ответ возвращается
- "/reset" сбрасывает сессию
- Reply включает оригинальное сообщение в контекст
- Незарегистрированный пользователь не получает ответа
- Placeholder "Барыга ищет товар..." показывается сразу

- Файл от пользователя скачивается во временную директорию

### Ожидаемые файлы
- `app/handlers/files.py`
- `app/gateway/tools.py` (заглушка)

### Технологии

### Ограничения

- TrueConf может иметь лимиты на размер файлов — заложить проверку (макс 10 МБ)

## Приложения

### А. API Gateway — подтверждённые и неподтверждённые endpoints

| Endpoint | Статус | Примечание |
|----------|--------|-----------|
| `POST /v1/chat/completions` | ✅ Подтверждён | Основной метод отправки сообщений |
| `GET /v1/models` | ✅ Подтверждён | Список моделей |
| `POST /tools/invoke` | ⚠️ Ограничен | `sessions_send` в deny list |
| Сброс сессии | ❓ Требует уточнения | Нет явного endpoint для reset через HTTP |

### Б. Риски и открытые вопросы

1. **Файловый API** — как передавать файлы через `/v1/chat/completions`? Нужна проверка.
2. **Streaming** — поддерживает ли Gateway streaming с tool events через `/v1/chat/completions`? Нужна проверка.
3. **Session reset** — как сбросить сессию через HTTP API? Возможный вариант: удалить session key из SessionStore и создать новый.
4. **python-trueconf-bot стабильность** — v1.2.x, API может меняться.
5. **edit_text() rate limit** — TrueConf может ограничивать частоту редактирования сообщений.
