# cortex-memory — Технический анализ

**Источники:** изучены исходные коды `openclaw-cortex` (плагин, открытый) и `cortex-memory` (OpenClaw skill), а также API-типы из `client.ts`. Бэкенд `cortex-api` — приватный репозиторий Ubundi.

---

## Как работает (архитектура)

### Компоненты

| Компонент | Репозиторий | Роль |
|---|---|---|
| `openclaw-cortex` | github.com/ubundi/openclaw-cortex | OpenClaw плагин — встраивается в цикл обработки сообщений |
| `cortex-memory` SKILL | github.com/openclaw/skills/…/cortex-memory | Инструкция для агента — как пользоваться памятью |
| `cortex-api` | github.com/ubundi/cortex-api (**приватный**) | Бэкенд Ubundi — хранит и обрабатывает воспоминания |
| TooToo Bridge | часть cortex-api | Персональный профиль — ценности, убеждения, мечты |

### Цикл работы

```
Пользователь → OpenClaw → cortex plugin (capture)
                                 ↓
                          Cortex API (Ubundi cloud)
                                 ↓
                          Knowledge Graph
                                 ↓
Пользователь → OpenClaw → cortex plugin (recall)
                                 ↓
                    <cortex_memories> block
```

**Plugin hooks в openclaw-cortex:**
- `beforeAgentStart` → **auto-recall** (до генерации ответа)
- `agentEnd` → **auto-capture** (после завершения ответа)
- `afterToolCall` → логирование аудита
- Separate handlers: bridge (TooToo), tools (cortex_search_memory, cortex_save_memory и т.д.)

### API endpoints (из client.ts)

```

/v1/ingest            — ingest одного текста
/v1/ingest/conversation — ingest массива сообщений
/v1/ingest/batch     — batch ingest
/v1/jobs/ingest       — async submit
/v1/jobs/reflect      — запуск фонового reflection
/v1/warmup            — warmup tenant
/v1/remember          — agent-side remember (async)
/v1/recall            — agent-side recall
/v1/forget/session/:id — удалить по session
/v1/forget/entity/:name — удалить по entity
/v1/forget/user/:id   — full GDPR delete
/v1/knowledge         — stats: total_memories, total_sessions, maturity
/v1/stats             — pipeline_tier (1|2|3), pipeline_maturity
/v1/nodes/:id         — детали ноды + related nodes + confidence_explanation
/v1/auth/link         — TooToo link status
/v1/auth/code         — pairing code для TooToo
/v1/bridge/qa         — отправка Q&A в TooToo профиль
/health
```

## auto-capture — что и как вытаскивает

### Trigger: событие `agentEnd`

После каждого ответа агента плагин:
1. Собирает **весь transcript** сессии (все сообщения user + assistant)
2. Прогоняет через **фильтры** (см. ниже)
3. Отправляет в `POST /v1/ingest/conversation`
4. API возвращает `IngestResponse` с созданными нодами и рёбрами

### Фильтрация ДО отправки (client-side, `filter.ts`)

Это **критически важно** для privacy — перед отправкой вырезается:

**Целиком удаляемые сообщения (низкий сигнал):**
- `HEARTBEAT_OK`, статусные сообщения ("status: idle", "connected | idle")
- Timestamp-only строки, декоративные линии (`─`, `═`)
- OpenClaw session startup boilerplate
- Filesystem metadata chatter ("File ... has permissions drwxr-xr-x")
- Audit log status lines

**Удаляемые из текста сообщения блоки:**
- Runtime metadata wrappers (`<channel info>`, `<sender info>`) от Telegram/messaging
- Инъецированные Cortex-блоки (`<cortex_memories>`, `<cortex_recovery>`)
- `[replying to ...] /[replying]`, `[quoting ...] /[quoting]` metadata

**Вырезаемые Sentence-level паттерны (volatile state):**
- Версии: `version 2.5.0`, `running node 20.11`
- Task status: `task X is in progress`, `currently working on`
- Ephemeral runtime: `port 8080`, `PID 1234`
- Deploy/build status: `deploy is running`, `CI is failing`
- Temporal markers: `right now`, `at the moment`, `as of now`

### Что отправляется в API: `IngestResponse`

После обработки API возвращает:

```typescript
interface IngestResponse {
  facts: IngestFact[];        // FACT nodes
  entities: IngestEntity[];   // ENTITY nodes
  emotions: EmotionOut[];     // EMOTION nodes
  values: ValueOut[];          // VALUE nodes
  beliefs: BeliefOut[];        // BELIEF nodes
  insights: InsightOut[];      // INSIGHT nodes
  life_context: LifeContextOut; // LIFECONTEXT node
  nodes_created: number;
  edges_created: number;
  duplicates_merged?: number;
}
```

**Поля provenance (отправляемые с каждым запросом):**
```typescript
{
  user_id: string;           // ОБЯЗАТЕЛЬНО — идентификатор пользователя
  source_origin: "openclaw"; // откуда: openclaw, tooooto, etc.
  derivation_mode: "inferred"; // inferred | stated | observed
  source_app: "OpenClaw";
  session_id?: string;
  source_channel?: string;    // telegram, discord, etc.
  origin_session_id?: string;
  session_goal?: string;
}
```

## Какие данные уходят в облако

### Итого: полный transcript + metadata

**Уходит:**
1. **Все user-сообщения** (после sanitization)
2. **Все assistant-ответы** (после sanitization)
3. **user_id** — глобальный идентификатор пользователя
4. **session_id** — идентификатор сессии
5. **source_channel** — канал (telegram, discord, etc.)
6. **session_goal** — цель сессии (если задана через `cortex_set_session_goal`)
7. **reference_date** — timestamp для temporal reasoning

**Фактически НЕ уходит (отфильтровано):**
- Tool outputs (инструменты не captured by default)
- Secrets, credentials, API keys
- Volatile state (версии, порты, PID, task status)
- Heartbeat loops
- Runtime metadata wrappers
- Инъецированные `<cortex_memories>` блоки

### TooToo Bridge — дополнительный поток

Если пользователь привязал агента к TooToo:
- Обнаруживаются **рефлексивные вопросы** через regex-паттерны
- Q&A пары отправляются в `/v1/bridge/qa`
- Target sections: `coreValues`, `beliefs`, `principles`, `ideas`, `dreams`, `practices`, `shadows`, `legacy`
- Это **личные ценности, страхи, мечты** — уходят в Ubundi cloud

## Self-hosted — возможно ли?

**Короткий ответ: нет.**

- `cortex-api` — приватный репозиторий Ubundi
- Нет упоминания self-hosted в коде или документации
- API key от Ubundi hardcoded в конфигурации плагина
- Endpoint URL по умолчанию: `https://api.cortex.ubundi.com` (или аналогичный)
- Адаптивные pipeline tiers (1|2|3) — бэкенд сам решает какой pipeline запускать
- Pipeline tier определяет качество retrieval (graph traversal, reranking)

**Возможные обходные пути (не поддерживаются):**
1. Развернуть свой Cortex-совместимый API (документация закрыта, только по type definitions из client.ts)
2. Локальный форк плагина с заменой baseUrl на свой endpoint (но тогда UI/TooToo bridge не работают)
3. Отключить auto-capture, использовать только `cortex_save_memory` с локальным хранилищем — но это не реализовано в плагине

**Cortex без облака технически невозможен** — плагин спроектирован вокруг Ubundi API.

## Безопасность и privacy

### Что Ubundi хранит на своих серверах

| Категория | Хранится? | Детали |
|---|---|---|
| Полные transcripts | ✅ Да | После клиентской фильтрации |
| user_id | ✅ Да | Привязка к аккаунту Ubundi |
| Session IDs | ✅ Да | Связывание по сессиям |
| Source channel | ✅ Да | Telegram, Discord, etc. |
| FACT nodes | ✅ Да | Извлечённые факты с timestamps |
| ENTITY nodes | ✅ Да | Именованные сущности |
| EMOTION nodes | ✅ Да | Эмоции + контекст |
| VALUE/BELIEF nodes | ✅ Да | Из TooToo bridge |
| API keys/credentials | ❌ Нет | Фильтруются на клиенте |
| Tool outputs | ❌ По умолчанию нет | Только assistant/user messages |
| Volatile state | ❌ Нет | Вырезается на клиенте |

### Механизмы защиты (на клиенте)

**Фильтрация секретов (`filter.ts`):**
- VOLATILE_STATE_PATTERNS вырезают версии, порты, PIDs
- LOW_SIGNAL_PATTERNS вырезают известные noise-паттерны
- Secrets и credentials фильтруются capture pipeline

**Namespace isolation:**
- `user_id` обязателен для всех ingest операций
- API разделяет данные между пользователями

**Audit logging (опционально):**
- `/audit on` — все отправки в API логируются в `~/.cortex/audit/`
- index.jsonl + payload files (каждый запрос отдельно)
- Пользователь может посмотреть что exactly ушло в облако

**Пользовательский контроль:**
```typescript
// в config.ts
autoCapture: boolean;      // отключить auto-capture
autoRecall: boolean;      // отключить auto-recall
auditEnabled: boolean;    // включить аудит
```

**Команды для пользователя:**
- `cortex_forget` — удалить конкретные воспоминания (query-based)
- `cortex_forget` по entity — удалить все о сущности
- `/forget user` — полное удаление (`DELETE /v1/forget/user/:id`)
- `/checkpoint` — сохранить summary перед reset

### Риски и проблемы

1. **Полные transcripts уходят в ЮАР (Ubundi, Cape Town)** — юрисдикция EU GDPR может быть проблемой
2. **TooToo Bridge отправляет личные ценности** — это sensitive данные, отправляемые без отдельного consent flow
3. **API key от Ubundi** — требует регистрации у Ubundi, зависимость от third-party
4. **Pipeline tiers неизвестны** — непрозрачно как работает graph traversal и reranking
5. **Confidence scores непрозрачны** — backend сам решает quality retrieval
6. **Нет документации на API** — приватный репозиторий, только type definitions

## Graph и temporal reasoning — как устроено

### Node Types (graph entities)

```typescript
type NodeType = "FACT" | "ENTITY" | "EMOTION" | "INSIGHT" 
              | "VALUE" | "BELIEF" | "LIFECONTEXT" | "SESSION" 
              | "COMMUNITY";
```

**FACT** — основная единица памяти:
```typescript
interface IngestFact {
  core: string;           // текст факта
  fact_type: string;      // тип: "preference", "decision", etc.
  occurred_at: string | null;  // TIMESTAMP для temporal reasoning
  entity_refs: string[];  // связанные ENTITY
  speaker: string;        // "user" | "assistant"
}
```

**ENTITY** — именованные сущности с aliases:
```typescript
interface IngestEntity {
  name: string;
  type: string;
  aliases: string[];
}
```

**EMOTION** — эмоции с evidence quote:
```typescript
interface EmotionOut {
  name: string;          // "frustration", "excitement"
  intensity: string;     // "high", "medium", "low"
  context?: string;      // где в разговоре
  evidence_quote?: string;
}
```

### Graph Edges

Из `NodeDetailResponse`:
```typescript
interface EdgeOut {
  type: string;
  strength: number;      // 0-1, strength связи
  reason?: string;        // почему связаны
}
```

Edges создаются автоматически при ingest — связывание FACT с ENTITY, фактов между собой через shared entities и temporal proximity.

### Temporal Reasoning

**occurred_at** — каждая FACT нода имеет timestamp. Это позволяет:
1. Запрашивать воспоминания "as of date X" через `reference_date` параметр
2. Temporal decay в confidence — `confidence_explanation.temporal_decay`
3. Приоритизация свежих фактов при дубликации (isNewer в formatter.ts)
4. Исторический lookup для benchmark-раннеров через `[cortex-date: YYYY-MM-DD]` маркер

**Pipeline Tiers:**
```
Tier 1: flat retrieval (semantic search, no graph)
Tier 2: flat + reranking
Tier 3: flat + reranking + graph traversal
```

Клиент запрашивает `mode: "fast" | "full"`, backend сам решает какой tier использовать. `deriveEffectiveTimeout` в `recall/handler.ts`:
- Tier 1: обычный timeout
- Tier 2: ×1.5 (минимум 12s)
- Tier 3: ×2.0 (минимум 20s)

### Recall Profiles (temporal-aware query routing)

Из `context-profile.ts`:
```typescript
type RecallProfile = "default" | "factual" | "planning" | "incident" | "handoff";
```

**Profile inference по ключевым словам в запросе:**
- `INCIDENT_RE`: outage, incident, sev1-4, broken, failure → "incident"
- `HANDOFF_RE`: resume, continue, handoff, "where did i" → "handoff"
- `PLANNING_RE`: plan, architecture, roadmap, design → "planning"
- `FACTUAL_RE`: "what is", "which", "how many" → "factual"

**Каждый профиль имеет разные параметры:**
| Profile | queryType | limit | minConfidence | mode |
|---|---|---|---|---|
| incident | factual | ×2 | 0.3 | fast |
| handoff | combined | normal | — | fast |
| planning | combined | ×1.5 | — | fast |
| factual | factual | normal | 0.3 | **full** (reranking!) |
| default | config | normal | — | config |

### Confidence Scoring

Из `formatMemoriesWithStats`:
```typescript
// query_alignment — насколько memory match запросу
< 0.4: "[weak match — do not cite specifics]"
< 0.6: "[topic match — verify details]"
>= 0.6 + score >= 0.6: "trusted"

// temporal_decay — penalize старые факты
confidence_explanation: {
  base_confidence: number;
  supporting_nodes: number;
  contradicting_nodes: number;
  temporal_decay: number;   // ← temporal penalty
  final_confidence: number;
}
```

## Выводы

### Плюсы

- **Серьёзная инженерия** — многослойная фильтрация, adaptive timeouts, pipeline tiers, deduplication, noise filtering. Качество продуманное.
- **Temporal reasoning** — graph + timestamps + reference_date дают осмысленный time-aware поиск
- **Recall profiles** — автоматический routing по типу запроса (incident vs planning vs factual)
- **Confidence scoring** — прозрачная система оценки качества с query_alignment
- **User controls** — forget по entity/session/user, audit logging, disable auto-capture/recall
- **TooToo bridge** — интересная идея personal knowledge graph (ценности, страхи, мечты)

### Минусы и риски

- **Vendor lock-in** — полная зависимость от Ubundi cloud, self-hosted невозможен
- **Privacy** — полные transcripts (после фильтрации) хранятся у Ubundi в ЮАР
- **TooToo privacy** — личные ценности и страхи уходят в third-party cloud
- **Closed source API** — непрозрачность backend, pipeline tiers undocumented
- **GDPR compliance** — unclear, юрисдикция ЮАР, нет публичной политики privacy
- **API key management** — зависимость от Ubundi как провайдера

### Для проекта unity-knowledge-base

Если цель — хранить **факты и решения** по проектам Романа (1С, вайбкодинг):
- Cortex фиксирует и извлекает факты хорошо
- Temporal reasoning полезен для "когда приняли это решение"
- Но: данные уходят в Ubundi (юрисдикция ЮАР)
- Self-hosted варианта нет
- Альтернатива: использовать встроенный OpenClaw LCM + файловую память, которая stays local

**Рекомендация:** использовать Cortex для долгосрочной memory, но с пониманием что данные в облаке Ubundi. Для sensitive данных (пароли, внутренние решения) — явный `cortex_save_memory` с отключённым auto-capture на уровне проекта.
