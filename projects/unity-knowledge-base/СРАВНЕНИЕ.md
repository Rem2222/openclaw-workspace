# Сравнительный анализ: unity-knowledge-base vs аналоги

## 🎯 Наша задача (что мы реально решаем?)

**Конкретная проблема Романа:**

LCM (Lossless-Claw) работает только для новых сессий. Исторические данные (все сессии до 23 марта) — не захвачены и недоступны через поиск. При каждом новом разговоре модель начинает "с нуля" — нет быстрого доступа к тому, что уже обсуждалось, решалось, делалось в предыдущих сессиях.

Дополнительно: файловая система Романа (`~/.openclaw/workspace/`) содержит сотни документов, конфигов, кода, заметок — но все они лежат мёртвым грузом. Чтобы найти что-то, нужен `grep` или ручной поиск. Никакого семантического поиска.

**Итого — две боли:**
1. Исторические сессии агента недоступны (LCM gap)
2. Файлы workspace не проиндексированы для семантического поиска

## brainrepo

### Описание

**Репозиторий:** `github.com/openclaw/skills/tree/main/skills/codezz/brainrepo`

Система персонального знания на базе PARA (Projects, Areas, Resources, Archives) + Zettelkasten. Хранит всё в `.md` файлах в Git-репозитории по адресу `~/Documents/brainrepo/`. Полностью файловая, без векторов, без БД, без облака.

Триггеры: "save this", "remember", "note", "capture", "brain dump", команды для ежедневного/еженедельного ревью.

### Архитектура

```
~/Documents/brainrepo/          ← Фиксированный путь (не конфигурируется)
├── Inbox/                      ← Быстрый захват (очищать каждый день)
├── Projects/                   ← Активные проекты с дедлайнами
├── Areas/                      ← Долгосрочные ответственности (без дедлайна)
├── Notes/                      ← Атомарные знания (Zettelkasten)
├── Resources/                  ← Внешние ссылки, статьи, книги
├── Journal/                    ← Ежедневные заметки (YYYY-MM-DD.md)
├── People/                     ← Один файл на человека
├── Tasks/                      ← Централизованный трекинг задач
└── Archive/                    ← Завершённые проекты
```

**Стек:** Чистый Markdown + Git. Никаких баз данных, никаких API, никаких external сервисов. Всё локально. Поиск — через grep/AG/вручную.

**Ключевые правила:**
- Dump → Process → Retrieve (сначала захватываешь ВСЁ в Inbox, потом вечером раскладываешь по местам)
- Notes используют `[[wiki-links]]` для связывания
- Ежедневный и еженедельный ревью — обязательная часть системы

### Что похоже на нашу концепцию

- Тема организации workspace — пересекается
- Концепция "capture → organize → retrieve" — та же трёхфазная модель
- Структурированное хранение с категориями (Projects, Areas, Notes)

### Что отличается

| brainrepo | unity-knowledge-base |
|-----------|---------------------|
| 100% файловая система | Vector DB (Qdrant) |
| Ручной поиск (grep) | Семантический поиск по эмбеддингам |
| Нет автоматизации | File watcher + автоиндексация |
| Нет LLM-интеграции | LLM может использовать как RAG |
| Человек сам раскладывает по папкам | Индексация автоматическая |
| Не захватывает сессии | Бэкфил исторических сессий |

### Что лучше у них

1. **Простота** — не нужен Docker, не нужна модель эмбеддингов, не нужен API-ключ. Сел и работай.
2. **Версионирование** — Git-история всех изменений из коробки. Можно откатиться, посмотреть когда что-то изменилось.
3. **Совместимость** — открывается в Obsidian, VS Code, любом редакторе. Никакого вендор-локапа.
4. **Privacy** — данные лежат локально, никуда не уходят. 100% суверенитет.
5. **Понятная структура** — PARA методология проверена тысячами людей. Не нужно изобретать велосипед.

### Что лучше у нас

1. **Семантический поиск** — brainrepo не умеет искать по смыслу. Только grep по словам.
2. **Автоматизация** — человеку не нужно руками раскладывать файлы. Индексатор делает это сам.
3. **Бэкфил сессий** — возможность захватить историю разговоров и сделать её доступной для поиска.
4. **RAG-режим** — модель может получать релевантные файлы в контекст автоматически.

## cortex-memory

### Описание

**Репозиторий:** `github.com/openclaw/skills/tree/main/skills/matthewubundi/cortex-memory`  
**Плагин:** `github.com/ubundi/openclaw-cortex`  
**Провайдер:** Ubundi (cloud service, MIT лицензия)

Система долгосрочной памяти для OpenClaw агентов. Автоматически захватывает факты из разговоров, хранит в облаке (Cortex API), предоставляет поиск и инструменты для явного сохранения/забывания.

### Архитектура

```
┌──────────────────────────────────────────────────────────────┐
│                     OpenClaw агент                           │
├──────────────────────────────────────────────────────────────┤
│  <cortex_memories>  ←  Auto-recall (перед каждым turn)     │
│                                                              │
│  cortex_search_memory()   ←  Инструмент для LLM             │
│  cortex_save_memory()     ←  Инструмент для LLM             │
│  cortex_forget()          ←  Инструмент для LLM             │
│  cortex_set_session_goal()←  Установить цель сессии         │
└──────────────────────────────────────────────────────────────┘
                          ↕ HTTPS (Cortex API)
                          ↓
┌──────────────────────────────────────────────────────────────┐
│                    Cortex API (cloud)                         │
│  - Knowledge Graph (сущности, связи, temporal reasoning)     │
│  - Factual recall, decisions, preferences, recent memory   │
│  - Namespace isolation per userId                           │
│  - Auto-capture: факты из conversation transcripts          │
│  - Session goals: bias recall, tag captures                │
└──────────────────────────────────────────────────────────────┘

Локально (openclaw-cortex plugin):
  - ~/.openclaw/cortex-user-id       ← Уникальный ID инсталла
  - ~/.openclaw/extensions/         ← Плагин
  - workspace/.cortex/audit/         ← Локальные аудит-логи
```

**Стек:**
- **Хранение:** Cortex API (Ubundi cloud) — knowledge graph с сущностями, связями, temporal reasoning
- **Клиент:** `@ubundi/openclaw-cortex` plugin для OpenClaw
- **Интерфейс:** LLM-инструменты (`cortex_search_memory`, etc.) + CLI (`openclaw cortex search`)
- **Синхронизация:** Автоматический sync `MEMORY.md` и daily logs в фоне

**Триггеры и поведение:**
- `autoRecall: true` — перед каждым turn делается запрос к Cortex API, результаты в `<cortex_memories>` блоке
- `autoCapture: true` — после каждого turn transcript отправляется в Cortex для факт-экстракции
- Явный save: `cortex_save_memory` для сохранения конкретных фактов/решений
- Session goals: агент автоматически определяет цель сессии, это biases recall

### Что похоже на нашу концепцию

- Та же проблема: LLM не имеет доступа к историческим данным между сессиями
- Концепция долгосрочной памяти для агента — пересекается полностью
- RAG-режим (cortex_memory как augment для generation)

### Что отличается

| cortex-memory | unity-knowledge-base |
|---------------|---------------------|
| **Cloud (Cortex API)** — данные уходят на сторону | 100% локально (Qdrant + файлы) |
| **Только факты из разговоров** — не индексирует файлы | **Индексирует файлы** — документы, код, конфиги |
| **Knowledge Graph** — сущности, связи, temporal reasoning | **Vector DB** — семантический поиск по chunk-ам |
| **Нет file watcher** — только conversation capture | **File watcher** — автоиндексация при изменении файлов |
| **Plugin + API key** — нужен аккаунт Ubundi | **Self-hosted** — Qdrant в Docker, эмбеддинги локально |
| **autoRecall/autoCapture** — прозрачно для пользователя | Ручная команда `?знания [запрос]` |

### Что лучше у них

1. **Knowledge Graph** — cortex-memory понимает СВЯЗИ между фактами, temporal reasoning (когда что-то произошло), различает факты/решения/предпочтения. Vector DB unity-knowledge-base видит только "похожесть" документов.
2. **Автоматизм** — не нужно ничего делать. Auto-capture извлекает факты автоматически. Пользователю не надо думать о том, "а сохранил ли я это".
3. **Зрелость** — это продакшен-система (v1.7.5), за ней стоит компания Ubundi, есть CI/CD, тесты, документация.
4. **Entity tracking** — отслеживание конкретных сущностей (люди, проекты, инструменты) с историей изменений.
5. **Session goals** — умный bias recall в зависимости от текущей задачи.

### Что лучше у нас

1. **Локальность** — никаких облаков, никаких API-ключей, данные не покидают машину. Для Романа (1С-разработчик, параноик про данные?) это критично.
2. **Индексация файлов** — cortex-memory не умеет в файлы. Он захватывает только факты ИЗ РАЗГОВОРОВ. Все файлы workspace останутся невидимыми.
3. **Бэкфил старых сессий** — unity-knowledge-base может импортировать старые JSONL напрямую. Cortex API получает только то, что передаёт плагин — старые сессии (март 19-22) в него не попадут.
4. **No vendor lock-in** — если Ubundi закроется или изменит pricing, cortex-memory станет бесполезным. Qdrant останется.

## 🔮 Решение задачи

### Что реально решит проблему Романа

**Проблема 1: Исторические сессии недоступны**

- `unity-knowledge-base` — **решит частично**: нужен бэкфил JSONL → создаст embeddings → поиск по старым сессиям заработает
- `cortex-memory` — **НЕ решит**: он получает данные только через плагин, старые сессии в него не попадут
- `brainrepo` — **НЕ решит**: это файловая ORGANIZATION система, не search система

**Проблема 2: Файлы workspace не ищутся**

- `unity-knowledge-base` — **решит полностью**: file watcher + embeddings → семантический поиск по всем файлам
- `cortex-memory` — **НЕ решит**: он не индексирует файлы, только факты из разговоров
- `brainrepo` — **Частично**: человек может складывать файлы в структуру и искать grep-ом, но это ручная работа

**Вывод:**

| Проблема | unity-knowledge-base | cortex-memory | brainrepo |
|----------|---------------------|---------------|-----------|
| Исторические сессии | ✅ Да | ❌ Нет | ❌ Нет |
| Поиск по файлам | ✅ Да | ❌ Нет | ❌ Нет (grep only) |
| Автоматический захват | Частично | ✅ Полностью | ❌ Нет |

### Что останется нерешённым

1. **Связи между фактами** — unity-knowledge-base видит "похожие chunk-и", но не понимает причинно-следственные связи. Cortex-memory в этом сильнее.
2. **Версионирование** — ни unity-knowledge-base, ни cortex-memory не дают Git-истории изменений.
3. **Ручная работа** — в brainrepo человек сам раскладывает по папкам (есть минусы: может забить; есть плюс: заставляет думать об организации).

### Стоит ли делать проект или взять готовый аналог?

**brainrepo** — не аналог. Это другая задача (PKM/organization), не search. Не подходит.

**cortex-memory** — не решает задачу Романа напрямую: он не индексирует файлы. Но! Он решает СМЕЖНУЮ проблему — захват фактов из текущих и будущих разговоров. Это ценная система сама по себе, но она НЕ заменяет unity-knowledge-base.

**unity-knowledge-base** — делать. Это единственное решение, которое реально индексирует файлы и исторические сессии.

**Оптимальный стек:**
- `unity-knowledge-base` — для файлов и исторических сессий
- `cortex-memory` — для автоматического захвата фактов из текущих и будущих разговоров (дополнение)

## 📋 Рекомендация

### Делать unity-knowledge-base, но с оговорками

**Аргументы ЗА:**
1. Это единственное решение, которое реально решит обе проблемы Романа
2. Локальное, no vendor lock-in, данные не уходят
3. Qdrant + sentence-transformers — зрелые технологии, не rocket science

**Аргументы ПРОТИВ или "подождать":**
1. **cortex-memory** уже делает half of the job — захват фактов из разговоров. Возможно, стоит сначала поставить cortex-memory (5 минут + API key), а потом уже строить unity-knowledge-base?
2. **Сложность** — Qdrant требует Docker, Python indexer, sentence-transformers (CPU, медленно). Это не "поставил и забыл".
3. **Бэкфил LCM** уже идёт как subagent. Результат покажет, насколько ценен этот подход.

### Что делать дальше

**Шаг 1 (немедленно):** Установить `cortex-memory` как дополнение
```bash
openclaw plugin install @ubundi/openclaw-cortex@latest
openclaw config set tools.profile full
# Добавить apiKey в конфиг
openclaw gateway restart
```
Это решит проблему захвата новых разговоров практически мгновенно.

**Шаг 2 (пока работает бэкфил LCM):** Понять, насколько полно бэкфил захватывает историю. Если хорошо — часть задачи уже решена.

**Шаг 3 (после бэкфила):** Запустить unity-knowledge-base — сначала MVP с ручным импортом JSONL в Qdrant, потом file watcher.

### Не делать

- **brainrepo** как замену — это не search система, это organization система. Не решает задачу.
- **cortex-memory вместо** unity-knowledge-base — не индексирует файлы, это другое.
- **Ждать идеального решения** — делать параллельно с бэкфилом.
