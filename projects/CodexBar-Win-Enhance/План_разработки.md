# План разработки — Добавление z.ai провайдера

## Этапы

### Этап 1: Анализ существующего кода
**Субагент:** coder (GLM-5)
**Цель:** Изучить ClaudeDataFetcher и UI-структуру codexbar.py для точной аналогии
**Выход:** Понимание паттернов: как регистрируются fetcher'ы, как строятся табы, как хранятся настройки
**Оценка времени:** 10-15 минут

#### Шаги:
1. Прочитать codexbar.py, найти класс ClaudeDataFetcher
2. Найти место регистрации fetcher'ов в главном цикле
3. Найти код UI-табов и настроек
4. Задокументировать найденные паттерны в комментариях к плану

### Этап 2: Класс ZaiDataFetcher
**Субагент:** coder (GLM-5)
**Цель:** Реализовать fetcher для z.ai API
**Выход:** Рабочий класс ZaiDataFetcher в codexbar.py
**Оценка времени:** 15-25 минут

#### Шаги:
1. Создать класс `ZaiDataFetcher` по аналогии с `ClaudeDataFetcher`
2. Реализовать метод `_fetch_from_api()`:
   - GET `https://api.z.ai/api/monitor/usage/quota/limit`
   - Header: `Authorization: Bearer {token}`
   - Парсинг JSON: quota, used, remaining, reset_date
3. Реализовать хранение/чтение API token (env var `ZAI_API_TOKEN` или config файл)
4. Обработка ошибок: таймаут, невалидный токен, нет сети

### Этап 3: Интеграция в главный цикл
**Субагент:** coder (GLM-5)
**Цель:** Подключить ZaiDataFetcher к циклу обновления данных
**Выход:** z.ai данные обновляются наравне с Claude/OpenAI
**Оценка времени:** 5-10 минут

#### Шаги:
1. Добавить экземпляр ZaiDataFetcher в список fetcher'ов
2. Убедиться что результаты попадают в общий data-словарь
3. Проверить что ошибки одного fetcher'а не ломают другие

### Этап 4: UI — таб z.ai
**Субагент:** coder (GLM-5)
**Цель:** Добавить вкладку с отображением z.ai данных
**Выход:** Визуальный таб с quota/used/remaining/reset_date
**Оценка времени:** 15-25 минут

#### Шаги:
1. Добавить вкладку «z.ai» в UI по аналогии с существующими табами
2. Отображать: quota, used, remaining, reset_date
3. Добавить прогресс-бар (used/quota)
4. Цветовая индикация при приближении к лимиту

### Этап 5: Настройки — ввод API token
**Субагент:** coder (GLM-5)
**Цель:** UI для ввода и сохранения z.ai API token
**Выход:** Поле ввода в настройках, токен сохраняется в config
**Оценка времени:** 10-15 минут

#### Шаги:
1. Добавить секцию «z.ai» в окно настроек
2. Поле ввода API token (masked)
3. Кнопка «Test» — проверить токен запросом к API
4. Сохранение токена в конфиг (зашифрованное или plaintext — как у других провайдеров)

### Этап 6: Тестирование
**Субагент:** coder (GLM-5)
**Цель:** Проверить что всё работает
**Выход:** Протестированный функционал
**Оценка времени:** 10-15 минут

#### Шаги:

```
Этап 1 (анализ) → Этап 2 (fetcher) → Этап 3 (интеграция) → Этап 4 (UI) → Этап 5 (настройки) → Этап 6 (тесты)
```

## Итого: 65-105 минут

## Риски

- **API токен недоступен для тестирования** — нужен реальный z.ai токен от Романа
- **Отличия в структуре ответа API** — может отличаться от ожидаемого, потребуется адаптация парсера
- **Единый файл архитектура** — при ~1936 строк добавление ещё ~150-200 строк может затруднить поддержку; рассмотреть рефакторинг в модули позже
