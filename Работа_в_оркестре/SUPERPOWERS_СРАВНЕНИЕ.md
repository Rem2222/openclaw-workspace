# Superpowers vs Оркестратор — Сравнение и предложения

## СИЛЬНЫЕ СТОРОНЫ

| Критерий | Superpowers | Оркестратор |
|----------|-------------|-------------|
| **Архитектура** | Модульные скиллы, легко добавлять новые | Жёсткая структура, 8 специализированных агентов |
| **Дизайн** | Socratic brainstorming, вопросы по одному | Аналитик собирает требования, но менее формально |
| **Планирование** | 2-5 минутные таски с готовым кодом | Крупные этапы (Backend/Frontend/Testing) |
| **Реализация** | Subagent-driven с two-stage review | Субагенты для каждого этапа, но без ревью тасков |
| **Тестирование** | **TDD RED-GREEN-REFACTOR** (хардкодит в процесс) | Есть этап тестирования, но без TDD |
| **Ревью** | Spec compliance → Code quality (2-stage) | Один этап ревью ТЗ, нет ревью кода |
| **Debugging** | 4-фазный systematic debugging | Нет явного процесса debugging |
| **Git** | Worktrees для изоляции | Нет работы с ветками |
| **Регистр проектов** | Нет | АКТИВНЫЕ_ПРОЕКТЫ.md с跟踪 |
| **Model/Time tracking** | Нет | Да (модель + время на этап) |
| **Learning loop** | Нет | Анализ после каждого этапа |
| **Источники** | Нет | ИСТОЧНИКИ.md с документацией |
| **Платформы** | Claude Code, Cursor, Codex, OpenCode, Gemini CLI | Только OpenClaw |
| **Сложность** | Много скиллов, сложно освоить | Проще структура |

## ПРЕДЛОЖЕНИЯ ПО УЛУЧШЕНИЮ ОРКЕСТРАТОРА

### 1. **Добавить TDD** (HIGH PRIORITY)

Сейчас: Этап тестирования в конце.

**Предложение:** Добавить TDD-процесс в этап разработки:
```
RED → пишем тест который падает
GREEN → пишем минимальный код
REFACTOR → улучшаем код
```

**Как:** Добавить правило в `ОБЩИЕ_ПРАВИЛА.md` и в промты разработчиков.

---

### 2. **Two-stage ревью** (HIGH PRIORITY)

Сейчас: Ревью только ТЗ.

**Предложение:** После каждой таски:
1. **Spec compliance review** — соответствует ли кода спеку?
2. **Code quality review** — качество кода, стиль, безопасность

**Как:** Добавить этап ревью после каждого субагента-разработчика.

---

### 3. **Изоляция через worktrees** (MEDIUM)

Сейчас: Работа на main/master.

**Предложение:** Перед разработкой создавать отдельную ветку.

**Как:** Добавить правило в `ОБЩИЕ_ПРАВИЛА.md`.

---

### 4. **Systematic debugging** (MEDIUM)

Сейчас: Нет процесса.

**Предложение:** Добавить 4-фазный debugging:
1. **Reproduce** — воспроизвести баг
2. **Diagnose** — найти причину
3. **Fix** — исправить
4. **Verify** — убедиться что исправлено

**Как:** Создать скилл `debugging.md` и добавить в АГЕНТЫ/07_ТЕСТИРОВЩИК.md.

---

### 5. **Model selection по сложности** (LOW)

Сейчас: Для каждого этапа фиксированная модель.

**Предложение:**
- Mechanical tasks (2-3 файла, простой код) → дешёвая модель
- Integration tasks (много файлов) → стандартная модель  
- Architecture/design/review → мощная модель

**Как:** Добавить в `ОБЩИЕ_ПРАВИЛА.md`.

---

### 6. **Visual companion** (LOW)

Сейчас: Нет визуальной поддержки.

**Предложение:** Использовать canvas/browser для UI дизайна перед разработкой.

**Как:** Добавить правило в brainstorming этап.
